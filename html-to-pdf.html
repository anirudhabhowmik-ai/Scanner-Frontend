    from playwright.async_api import async_playwright

async def generate_pdf_from_url(url, options):
    async with async_playwright() as p:
        browser = await p.chromium.launch(args=["--no-sandbox"])
        page = await browser.new_page()
        
        # Set viewport to simulate different devices
        viewport_width = int(options["screen_size"])
        await page.set_viewport_size({
            "width": viewport_width, 
            "height": 2000
        })
        
        # Load the page and wait for all resources
        await page.goto(url, wait_until="networkidle")
        
        # Generate PDF with custom options
        pdf_kwargs = {
            "path": pdf_path,
            "margin": {k: options["margin"] for k in ["top", "bottom", "left", "right"]},
            "print_background": True
        }
    • Playwright - Headless browser automation from Microsoft

    🔧 How HTML to PDF Conversion Actually Works

    Unlike simple libraries that just strip HTML tags, Playwright actually runs a full browser engine:

    Step 1: Launch Headless Browser

    browser = await p.chromium.launch(args=["--no-sandbox"])
page = await browser.new_page()

    Playwright launches a real Chromium browser in the background - no GUI, just the rendering engine. The lawyer's evidence pages are rendered exactly as a human would see them.

    Step 2: Set Viewport Size (Responsive Design)

    viewport_width = int(options["screen_size"])
if viewport_width <= 768:  # Tablet/Mobile
    viewport_config["device_scale_factor"] = 2
await page.set_viewport_size(viewport_config)

    This simulates different devices. The developer tests his responsive designs at desktop (1920px), tablet (768px), and mobile (320px) sizes. The browser actually renders the page at those dimensions, triggering all CSS media queries.

    Step 3: Load and Wait

    await page.goto(url, wait_until="networkidle")

    networkidle waits until there are no network connections for at least 500ms. This ensures all images, fonts, and JavaScript are loaded. The consultant's client websites with heavy JavaScript? Fully rendered before PDF generation.

    Step 4: Inject CSS (Ad Blocking)

    await page.add_style_tag(content="""
    iframe[src*="doubleclick.net"],
    div.adsbygoogle,
    ins.adsbygoogle {
        display: none !important;
    }
""")

    This is CSS-based ad blocking - safe and reversible. The lawyer's court evidence has no distracting ads, but the page structure remains intact.

    Step 5: Calculate Page Height

    content_height = await page.evaluate('document.documentElement.scrollHeight')

if options["long_page"]:
    pdf_kwargs.update({
        "width": f"{viewport_width}px",
        "height": f"{int(content_height * 1.1)}px"
    })

    For "one long page" mode, we calculate the actual content height and generate a PDF tall enough to fit everything. No page breaks, just one continuous scroll.

    The "Missing Images" Mystery

    For uploaded HTML files, images would sometimes be missing. The issue was relative paths - the HTML expected images in the same folder, but Playwright was running in a temporary directory.

    The "Infinite Scroll" Problem

    Some pages with infinite scroll would never finish loading. Playwright's networkidle would wait forever as new content kept loading. 

    # Fixed: Add timeout and fallback
try:
    await page.goto(url, wait_until="networkidle", timeout=30000)
except TimeoutError:
    # Continue anyway - we have something
    await page.goto(url, wait_until="domcontentloaded")

    The "Viewport Height" Bug

    For long pages, setting a fixed viewport height would cut off content. The page would render only the first screenful.

    ❓ Technical Questions People Actually Ask

    "Why use Playwright instead of wkhtmltopdf or WeasyPrint?"

    wkhtmltopdf uses an old WebKit engine that doesn't support modern CSS (Flexbox, Grid). WeasyPrint renders HTML but can't execute JavaScript. Playwright uses the latest Chromium - it's the same engine as Chrome. The developer's responsive designs with CSS Grid? Rendered perfectly.

    "What's the difference between networkidle and domcontentloaded?" 

    // domcontentloaded - HTML parsed, DOM ready
// networkidle - no network activity for 500ms

    domcontentloaded is faster but might miss images loaded after page load. networkidle ensures everything is loaded - the researcher's articles with lazy-loaded images? networkidle catches them all.
