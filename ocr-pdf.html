    🧠 How OCR Actually Works (Under the Hood)

    OCR isn't magic - it's a series of algorithms working together. Here's exactly what happens when you upload a PDF:

    Step 1: PDF to Images 

    # ocrmypdf handles this internally
# It converts each PDF page to an image for Tesseract

    PDFs can contain both text and images. For scanned documents, each page is actually an image. The first step is extracting those images at the right resolution (300 DPI is ideal).

    Step 2: Image Preprocessing 

    # ocrmypdf applies these automatically:
- Deskew: Straighten crooked pages
- Despeckle: Remove noise/dots
- Threshold: Convert to black and white for better recognition
- DPI adjustment: Ensure minimum 300 DPI

    This step is crucial. A crooked or noisy image will give garbage OCR results. I spent weeks tuning these preprocessing parameters.

    Step 3: Character Recognition (Tesseract) 

    # Tesseract uses LSTM neural networks
# It looks at each character and its context

tesseract image.png stdout -l eng+fra --psm 6

    --psm 6 = Treat image as a uniform text block
    -l eng+fra = Use English + French language models

    Tesseract 4+ uses LSTM (Long Short-Term Memory) neural networks - the same technology used in speech recognition. It doesn't just match shapes; it understands context. "c1ear" is probably "clear" because that makes sense.

    Step 4: Building the Searchable PDF 

    # ocrmypdf creates a PDF with:
- Original scanned images (visible layer)
- Recognized text (hidden layer)
- Text positioned exactly where it appears

ocrmypdf.ocr(
    input_path,
    output_path,
    language="eng+fra",
    force_ocr=True,        # Always add text layer
    deskew=True,           # Auto-straighten
    optimize=1            # Balance quality/size
)

    The result: a PDF that looks identical to your scan, but now you can search, copy, and highlight text.

    🐛 The Hard Parts: Debugging OCR (What Went Wrong)

    The "PriorOcrFoundError" Nightmare

    For weeks, the tool would randomly fail with PriorOcrFoundError on some PDFs. Turns out, some PDFs already had hidden text layers (from previous OCR attempts), and ocrmypdf would refuse to overwrite them.

    The fix: Always use force_ocr=True to ignore existing text: 

    # ALWAYS force OCR, even if PDF claims to have text
ocrmypdf.ocr(
    input_path,
    output_path,
    language=language_string,
    force_ocr=True,  # This fixed it
    skip_text=False
)

    The Memory Limit Problem

    Railway (the hosting platform) has memory limits. Processing 100-page PDFs would sometimes crash. I had to add resource limits: 

    # Limit resource usage
ocrmypdf.ocr(
    ...,
    jobs=2,                    # Only use 2 CPU cores
    max_image_mpixels=3000,   # Don't process huge images
    tesseract_timeout=300      # Timeout after 5 minutes per page
)

    The Retry Logic

    Sometimes OCR would fail on the first attempt but succeed on the second with different parameters. I added a retry mechanism: 

    try:
    # First attempt with normal settings
    ocrmypdf.ocr(..., force_ocr=True, jobs=2)
except ocrmypdf.exceptions.PriorOcrFoundError:
    # Retry with different settings
    ocrmypdf.ocr(..., force_ocr=True, redo_ocr=True, jobs=1)

    🖼️ Image Preprocessing: Why It Matters

    OCR accuracy depends 90% on image quality. Here's what happens behind the scenes:

    Deskew (Straightening) 

    # ocrmypdf does this automatically
# It finds text lines and rotates until they're horizontal

    Even a 2-degree tilt can reduce accuracy by 10-15%. Deskew fixes that.

    Despeckle (Noise Removal)

    Scanned documents often have tiny dots, smudges, or background "noise." These confuse OCR because they look like parts of letters. Despeckle removes them.

    Thresholding (Binarization) 

    # Convert to pure black and white
# Otsu's method finds optimal threshold automatically
_, thresh = cv2.threshold(gray, 0, 255, cv2.THRESH_BINARY + cv2.THRESH_OTSU)

    This makes text pop against the background. Otsu's algorithm analyzes the image histogram and picks the threshold that best separates text from background.

    📊 Accuracy: What I Learned From Testing 1,000+ Documents

    98%

    Clean, 300 DPI, printed text

    85-90%

    Faded print, low contrast

    50-70%

    Handwriting or tiny fonts

    What Affects Accuracy (From My Testing)

    • Resolution: 300 DPI is perfect. Below 200 DPI, accuracy drops fast.
    • Font size: Below 8pt, Tesseract struggles.
    • Background: Solid white = best. Patterns/textures confuse it.
    • Language mix: English+Spanish works great. Adding too many languages (5+) reduces accuracy.
    • Handwriting: Tesseract wasn't trained for it. Use a dedicated handwriting recognizer.

    ⚙️ ocrmypdf Parameters (What Each One Does)

    force_ocr=True

    Forces OCR even if the PDF already has text. Without this, ocrmypdf might skip pages that appear to have text (even if it's garbage). I always enable it.

    deskew=True

    Automatically straightens crooked pages. Uses Hough transform to detect text lines and rotates until they're horizontal. Adds maybe 1 second per page but worth it.

    optimize=1

    Level 1: balance quality and file size. Level 0 = no optimization (huge files). Level 3 = maximum compression (might lose quality). I found 1 works best.

    jobs=2

    Number of CPU cores to use. Railway gives limited resources, so I limit to 2 to avoid crashes.

    tesseract_timeout=300

    If a page takes more than 5 minutes, abort. This prevents the server from hanging on corrupted documents.

    ❓ Technical Questions People Actually Ask

    "Why can't you OCR handwriting?"

    Tesseract (the OCR engine) is trained on printed text. Handwriting recognition requires neural networks trained on handwriting samples - completely different technology. If enough people ask, I might add a separate handwriting tool using TensorFlow, but it's a big project.

    "What's the file size limit?"

    50MB per file (server limit). For larger documents, split them first using the Split PDF tool.

    "How do you handle right-to-left languages like Arabic?"

    Tesseract supports RTL languages natively. When Arabic is selected, it processes the text in reverse order and outputs correctly. The hidden text layer in the PDF also respects RTL.

    "Why does it take longer for some PDFs?"

    Processing time depends on: number of pages, image resolution, preprocessing needed, and number of languages selected. A 100-page PDF at 300 DPI with 3 languages might take 2-3 minutes. That's normal - OCR is computationally expensive.
