    🔧 How PDF Table Extraction Actually Works

    PDFs don't store "tables" - they store text at specific coordinates. Table extraction algorithms have to figure out which text belongs together:

    Method 1: Tabula (Coordinate-Based)

    # Tabula looks at text positions and groups by Y coordinate (rows)
# and X coordinate (columns)
{
  "text": "$1,234.56",
  "top": 120.5,   # Y position from top
  "left": 450.2,  # X position from left
  "width": 65.3
}

    How it works: Tabula clusters text by Y-position (same line = same row), then sorts by X-position to determine columns. The accountant's bank statements with simple layouts work well here.

    Method 2: Camelot Lattice (Border-Based)

    # Camelot lattice detects table borders using computer vision
1. Convert PDF page to image
2. Detect line segments (Hough transform)
3. Find intersections to create cell boundaries
4. Group text into cells based on boundaries

    How it works: Lattice mode actually "sees" the table borders. It's like OCR for table structure. Perfect for bank statements that have printed lines between rows and columns.

    Method 3: Camelot Stream (Whitespace-Based)

    # Stream mode uses text positioning and whitespace gaps
Column boundaries = positions where large gaps appear
Row boundaries = lines with consistent Y coordinates

    How it works: For tables without borders, stream mode looks at the spacing between text. If there's a consistent gap at X=200px across multiple lines, that's probably a column boundary.

    📋 Why PDF to Excel Is Surprisingly Difficult

    PDFs weren't designed for data extraction. They were designed to look the same everywhere. Here's what makes table extraction challenging:

    1. No Semantic Structure

    HTML has <table>, <tr>, <td>. PDF just has text at coordinates. The algorithm has to guess what's a table and what's just text.

    2. Visually Aligned != Structurally Aligned

    Numbers might look aligned in the PDF, but they could be positioned slightly differently. One cell might be at X=100.2, the next at X=200.1 - that's a 99.9 pixel gap, so it's a column boundary.

    3. Multi-line Cells

    A cell with wrapped text spans multiple lines. The extractor needs to know that "Total" on line 1 and "Amount:" on line 2 belong in the same cell.

    4. Merged Cells

    A header that spans three columns should produce one cell in the header row, but the data below should still have three columns. Lattice mode handles this well.
