    🔧 How PDF Rotation Actually Works (The /Rotate Entry)

    PDF rotation isn't like rotating an image - it's a simple flag in the page dictionary. Here's exactly what happens:

    The /Rotate Entry 

    // PDF page dictionary with rotation
<< /Type /Page
   /Parent 4 0 R
   /MediaBox [0 0 612 792]
   /Rotate 90  <-- This is all it takes
   /Contents 6 0 R
>>

    The /Rotate entry tells the PDF viewer how many degrees to rotate the page (clockwise). Values can be 0, 90, 180, or 270. The actual page content isn't changed - just the display instruction.

    PyPDF2's rotate() Method 

    # PyPDF2 source code (simplified)
def rotate(self, angle):
    current = self.get("/Rotate", 0)
    new_angle = (current + angle) % 360
    if new_angle != 0:
        self[NameObject("/Rotate")] = NumberObject(new_angle)
    elif "/Rotate" in self:
        del self[NameObject("/Rotate")]

    The method reads the current rotation, adds the new angle, mod 360, and updates the dictionary. If the result is 0, it removes the entry entirely (default).

    📊 The Global Page Index - Tracking Pages Across Multiple Files

    The Problem

    When users upload multiple PDFs, each file has its own page numbering (Page 1 of File A, Page 1 of File B). But the rotations array from the frontend is a flat list covering all pages in order. How do we know which rotation belongs to which page?

    The Solution: Global Page Index 

    # Backend tracking across multiple files
page_index = 0
for pdf_file in files:
    reader = PdfReader(pdf_file)
    for page in reader.pages:
        # Get rotation for this global index
        rotate = rotations[page_index] if page_index < len(rotations) else 0
        if rotate:
            page.rotate(rotate)
        writer.add_page(page)
        page_index += 1  # Increment for next page

    The frontend generates rotations in the order pages appear in the grid. The backend reads files in the same order, and the global index ensures each page gets its correct rotation, even when mixing files.

    Example

    File A (3 pages): User rotates page 2 only → rotations[0]=0, rotations[1]=90, rotations[2]=0
    File B (2 pages): User rotates no pages → rotations[3]=0, rotations[4]=0
    Global index tracks: File A pages use indices 0-2, File B pages use indices 3-4.

    📐 The Scale Factor - Why 0.7 for Vertical Pages?

    The Problem

    // Without scaling, rotated pages overflow
canvas.style.transform = `rotate(${r}deg)`;  
// A portrait page rotated to landscape now
// has width > container width → overflow

    When you rotate a page 90°, its width becomes its original height. If the container is sized for portrait, the rotated landscape page will overflow horizontally.

    The Solution

    // Calculate scale factor based on orientation
const isVertical = newRotation === 90 || newRotation === 270;
const scale = isVertical ? 0.7 : 1;
canvas.style.transform = `rotate(${newRotation}deg) scale(${scale})`;

    Why 0.7? For a typical page, width:height ratio is about 1:1.414 (A4). When rotated 90°, the page's display width becomes 1.414 times the container width. Scaling by 1/1.414 ≈ 0.707 fits it perfectly. 0.7 is close enough for visual preview.

    📏 Different Page Sizes

    The 0.7 factor works for most standard page sizes (A4, Letter, Legal). For unusually shaped pages, it might not be perfect, but it's good enough to identify content. The actual PDF rotation doesn't need scaling - viewers handle it automatically.

    🐛 The Debugging Stories (What Went Wrong)

    The "Rotation Accumulation" Bug

    Early versions sent the total rotation from the frontend (0, 90, 180, 270). But when users rotated a page multiple times, the backend would apply the new rotation on top of any existing rotation in the PDF. Pages would rotate twice. 

    // Wrong - assumes PDF has no existing rotation

    page.rotate(rotate)  // Adds to existing

// Fixed - PyPDF2's rotate() handles accumulation
// It reads current /Rotate and adds the new angle
// So sending total rotation is actually correct!

    The bug wasn't in the code - it was in my understanding. PyPDF2's rotate() adds to existing rotation, so sending the total (what the user sees) is exactly right. The office manager's rotated pages were always correct.

    The "Scale Factor" Overflow

    Initially, I didn't scale rotated pages at all. Landscape pages would overflow the card and overlap with the next page. The grid broke. 

    // Fixed: calculate scale based on rotation
const scale = (r === 90 || r === 270) ? 0.7 : 1;
canvas.style.transform = `rotate(${r}deg) scale(${scale})`;

    The "Multiple Files" Indexing Error

    When users uploaded multiple files, the rotations array was being reset per file. Page 1 of File B was getting rotation intended for page 1 of File A. 

    // Wrong - resetting index per file
for pdf_file in files:
    for i, page in enumerate(reader.pages):
        rotate = rotations[i]  // Wrong! Uses file-local index

// Fixed - global index
page_index = 0
for pdf_file in files:
    for page in reader.pages:
        rotate = rotations[page_index]  // Correct global index
        page_index += 1

    ❓ Technical Questions People Actually Ask

    "Is rotation lossless?"

    Yes. PDF rotation doesn't re-encode the page content. It just sets a flag in the page dictionary that tells viewers to display it rotated. Text remains searchable, images remain sharp, form fields still work. The accountant's rotated tax forms still had all his numbers.

    "What about rotated pages that were already rotated?"

    PyPDF2's rotate() adds to existing rotation. If a page already has /Rotate 90 and you rotate another 90, it becomes 180. The frontend's data-rotation tracks the total, so what you see matches what you get.

    "Why 0.7 scale for rotated pages?" 

    // For A4 page: width:height ≈ 1:1.414
// When rotated 90°, width becomes 1.414 × container
// Scale = 1/1.414 ≈ 0.707

    The 0.7 factor fits rotated pages in the container without overflow. It's an approximation, but good enough for preview.

    "Can I rotate only some pages?"

    Yes. Each page has its own rotate button. The rotations array tracks per-page rotation. The office manager rotates only the upside-down pages from her scanner, leaving the rest alone.

    "What about password-protected PDFs?"

    PyPDF2 can't open encrypted files. Use the Unlock PDF tool first, then rotate. The office manager learned this when she tried to rotate a password-protected company document.
