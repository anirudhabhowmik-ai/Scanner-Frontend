    Client-side Splitting (pdf-lib + JSZip) - Files under 20MB 

    // Small files split in YOUR browser - never leave your computer
async function splitFrontend(file, range) {
    const zip = new JSZip();
    const pdfBytes = await file.arrayBuffer();
    const pdfDoc = await PDFLib.PDFDocument.load(pdfBytes);
    
    // Parse user's page range (e.g., "1-3,5,7-9")
    const pages = parseRange(range, pdfDoc.getPageCount());
    
    for (const i of pages) {
        const newPdf = await PDFLib.PDFDocument.create();
        const [page] = await newPdf.copyPages(pdfDoc, [i]);
        newPdf.addPage(page);
        
        const bytes = await newPdf.save();
        zip.file(`page-${i+1}.pdf`, bytes);
    }
    
    return await zip.generateAsync({ type: "blob" });
}
    • pdf-lib - Creates new PDFs for each page by copying from original
    • JSZip - Packages multiple PDFs into one downloadable ZIP file
    • parseRange() - Custom parser that converts "1-3,5" into [0,1,2,4] (zero-indexed)

    🔧 How PDF Splitting Actually Works (The Page Object Model)

    Splitting a PDF isn't just "cutting" - it's creating new PDFs that each contain one page from the original. Here's what happens:

    Step 1: Reading the Page Tree 

    # Every PDF has a page tree structure
Pages Object
├── Kids [ 3 0 R, 4 0 R, 5 0 R ]  # References to page objects
├── Count 3                        # Total pages

# pypdf traverses this tree to find all pages
reader = PdfReader(file)
print(f"Total pages: {len(reader.pages)}")

    The PDF doesn't store pages in a simple list - they're in a tree structure for efficiency. When you ask for page 50, the reader navigates this tree to find it.

    Step 2: Creating New PDF Documents 

    // pdf-lib: Create a new empty PDF
const newPdf = await PDFLib.PDFDocument.create();

// Copy a page from the original
const [page] = await newPdf.copyPages(originalPdf, [pageIndex]);
newPdf.addPage(page);

    copyPages() is doing heavy lifting - it clones the page object AND all its dependencies (fonts, images, annotations). Each new PDF gets its own copy of everything it needs.

    Step 3: Resource Management 

    # Each page might reference shared resources
Page 1 Resources: Font-A, Image-X, Image-Y
Page 2 Resources: Font-A, Image-X, Image-Z

# When splitting, each new PDF gets its own copies:
page-1.pdf: Font-A, Image-X, Image-Y
page-2.pdf: Font-A, Image-X, Image-Z

    This is why splitting can increase total file size - each page gets its own copy of shared resources. The accountant's receipt scans? Each page had the same fonts - after splitting, each PDF was slightly larger than the original page's share.

    Step 4: ZIP Packaging 

    // JSZip compresses multiple PDFs into one file

zip.file("page-2.pdf", pdfBytes2);
const zipBlob = await zip.generateAsync({ type: "blob" });

    🔢 How Page Range Parsing Works

    The Range Parser Algorithm 

    function parseRange(range, totalPages) {
    // Example: "1-3,5,7-9" → [0,1,2,4,6,7,8] (zero-indexed)
    const pages = new Set();
    
    range.split(',').forEach(part => {
        if (part.includes('-')) {
            // Handle range like "1-3"
            let [start, end] = part.split('-').map(n => parseInt(n) - 1);
            for (let i = start; i <= end; i++) {
                if (i >= 0 && i < totalPages) pages.add(i);
            }
        } else {
            // Handle single page like "5"
            const p = parseInt(part) - 1;
            if (p >= 0 && p < totalPages) pages.add(p);
        }
    });
    
    return [...pages].sort((a,b) => a - b);
}

    Zero-indexing: Computers count from 0, but humans count pages from 1. The parser subtracts 1 from every number. "Page 1" in human language is index 0 in code.

    Examples of Range Parsing

    User Input Parsed Indices Pages Extracted
    "1-3" [0,1,2] 1,2,3
    "5,7,9" [4,6,8] 5,7,9
    "1-3,5,7-9" [0,1,2,4,6,7,8] 1,2,3,5,7,8,9
    "10-15,20" [9,10,11,12,13,14,19] 10,11,12,13,14,15,20

    🐛 The Debugging Stories (What Went Wrong)

    The "Off-by-One" Nightmare

    In early versions, typing "1-3" would give you pages 2,3,4. I forgot that humans count from 1, computers count from 0. The parser was subtracting one, but then the PDF library was also subtracting one. Double off-by-one. 

    // Fixed: Subtract once in parser, use directly in library
const p = parseInt(part) - 1;  // Human 1 → Computer 0
// Then pass directly to copyPages()

    The ZIP Memory Blow-up

    Splitting a 500-page document into 500 individual PDFs and then zipping them all at once would crash the server. JSZip and Python's zipfile were trying to hold everything in memory.

    The fix: Stream pages to ZIP as they're created, don't store all PDFs in memory: 

    # Python: Write each PDF to ZIP immediately
for i, page in enumerate(reader.pages):
    writer = PdfWriter()
    writer.add_page(page)
    pdf_bytes = io.BytesIO()
    writer.write(pdf_bytes)
    z.writestr(f"page-{i+1}.pdf", pdf_bytes.getvalue())
    # Memory freed after each iteration

    The "Page 0" Bug

    A user typed "0" in the range box. The parser tried to subtract 1 (getting -1), then pdf-lib tried to access page -1. Crash. 

    // Added bounds checking
if (p >= 0 && p < totalPages) pages.add(p);

    Now invalid pages are silently ignored instead of crashing.

    📦 Why ZIP? The Download Problem

    The Browser Limitation

    When you split a 50-page PDF, you get 50 separate files. Browsers have a limitation: you can't trigger 50 downloads at once. Each would pop a "Save As" dialog, and the browser would likely block them as spam.

    ZIP solves this: Package all 50 files into one container, trigger one download. The accountant who splits 30 receipts into individual files gets one ZIP, unzips it, and has all his expense report PDFs.

    ZIP Compression Levels 

    // JSZip: DEFLATE compression (like gzip)

    type: "blob",
    compression: "DEFLATE",
    compressionOptions: { level: 6 }  // 1-9, 6 = balance
});

    Level 6 is the sweet spot - good compression without slowing down. The 50-page split compressed 15% smaller than the raw PDFs.

    ❓ Technical Questions People Actually Ask

    "Why does splitting sometimes increase file size per page?"

    Shared resources. The original PDF might have a font used on 100 pages, stored once. After splitting, each page gets its own copy of that font. That 10KB font becomes 10KB × 100 pages = 1MB extra. The accountant's receipt scans had this issue - each page's file was slightly larger than 1/100th of the original.

    "What's the difference between splitting and extracting?"

    Splitting creates multiple files from one PDF (page-1.pdf, page-2.pdf, etc.). Extracting pulls specific pages into a new single PDF. This tool does splitting (multiple files). For extraction, use the Extract Pages tool.

    "Can I split by bookmarks or sections instead of page numbers?"

    Not yet. That would require parsing the PDF's outline tree, which is more complex. If enough people ask, I'll add it. For now, page numbers are the only way.

    "Why 20MB cutoff for browser processing?"

    Browser memory limits. pdf-lib needs to load the entire PDF into memory to access pages. A 20MB PDF uses about 80-100MB of RAM during processing. Above that, browser tabs can crash. The 20MB threshold is conservative - keeps things stable for everyone.
