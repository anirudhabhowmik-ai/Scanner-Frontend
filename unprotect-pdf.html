    🔐 How PDF Decryption Actually Works (Under the Hood)

    PDF encryption isn't magic - it's math. Here's exactly what happens when you click "Unlock PDFs":

    Step 1: Read the Encryption Dictionary 

    # pikepdf reads the PDF's encryption dictionary
# It looks for entries like:
<< /Type /Encrypt
   /Filter /Standard
   /V 2                   % Algorithm version
   /R 3                   % Revision (AES-128)
   /Length 128            % Key length
   /O (owner_hash)        % Owner password hash
   /U (user_hash)         % User password hash
>>

    This dictionary tells us how the PDF is encrypted and what kind of protection it has. If there's no encryption dictionary, the PDF isn't protected at all.

    Step 2: Try Opening Without Password 

    # First, try with empty password
try:
    pdf = pikepdf.open(input_path, password="")
    # Success! This PDF has restrictions but no open password
except pikepdf.PasswordError:
    # Password required

    Many PDFs have "owner passwords" (restrictions) but no "user password" (open password). These can be opened without a password, but pikepdf can still detect and remove the restrictions.

    Step 3: Password Verification (Hashing) 

    # pikepdf takes your password and hashes it
# It compares the hash to the stored /U (user) hash
# If they match, it derives the encryption key

def verify_password(pdf, password):
    # This is handled internally by pikepdf
    # It uses the same algorithm Adobe Reader uses
    return pdf.is_encrypted_with_password(password)

    The password itself isn't stored in the file - only a hash. This is why forgotten passwords can't be recovered.

    Step 4: Decryption (AES/CBC) 

    # Once we have the correct key, pikepdf decrypts each stream
# The PDF is loaded into memory in plaintext
pdf = pikepdf.open(input_path, password=correct_password)

# Save without encryption
pdf.save(output_path, encryption=False)

    The encryption=False parameter tells pikepdf to write the PDF without any encryption dictionary. The result is a completely unlocked file.

    🔑 User Password vs Owner Password: The Technical Difference

    User Password (Open Password) 

    # In the encryption dictionary:
/U (user_password_hash)   % Required to open

# pikepdf check:
if pdf.encryption.get('/U'):
    print("This PDF requires a password to open")

    The /U entry stores a hash of the user password. Without matching this hash, the file won't decrypt. This is what locked me out of my own contract - I couldn't match the hash because I forgot the password.

    Owner Password (Permissions Password) 

    # In the encryption dictionary:
/O (owner_password_hash)   % Controls permissions
/P -44                     % Permission flags

# Permission flags (bitmask)
ALLOW_PRINTING = 4
ALLOW_MODIFY = 8
ALLOW_COPY = 16
ALLOW_MODIFY_ANNOTATIONS = 32

    The /P value is a bitmask that controls what you can do with the file. -44 (or 0xFFFFFFD4 in hex) means printing allowed, copying allowed, but no editing. The office manager's templates had these flags set.

    🔧 How pikepdf Handles Both 

    # pikepdf automatically detects which case applies
try:
    # Case 1: No password needed (restriction-only)

    # Success - we can remove restrictions
except pikepdf.PasswordError:
    # Case 2: Password required
    pdf = pikepdf.open(input_path, password=user_password)

    This is why the tool behaves differently depending on your file - it's not guessing, it's reading the actual PDF structure.

    🐛 The Debugging Stories (What Went Wrong)

    The "401 Password Required" Implementation

    Initially, if a file needed a password, the tool just failed with a generic error. Users had no idea what went wrong. I had to redesign the flow: 

    # Old way (bad)
return jsonify({"error": "Failed to open PDF"}), 400

# New way (good)
if passwords_needed:
    return jsonify({
        "error": "password_required",
        "message": f"These files need a password: {', '.join(passwords_needed)}"
    }), 401  # 401 = Unauthorized - triggers password prompt

    The frontend now checks for status 401 and shows the password input. The office manager now knows exactly which files need passwords.

    The QPDF/pikepdf Memory Issue

    Early versions loaded entire PDFs into memory. A 500MB PDF with high-res images would crash the server. The fix was to use pikepdf's built-in streaming: 

    # pikepdf is built on QPDF, which handles streaming
# It doesn't need to load everything into memory at once
pdf = pikepdf.open(input_path, password=pwd)  # Memory-efficient
pdf.save(output_path, encryption=False)       # Streams output

    The graduate student who emailed about a 200MB research paper? Processed perfectly.

    "What's the /P -44 in the encryption dictionary?" 

    # -44 in decimal = 0xFFFFFFD4 in hex
# Binary: 11111111111111111111111111010100
# This means:
# - Printing allowed (bit 2 = 1)
# - Copying allowed (bit 4 = 1)
